# 수익률 모의실험(백테스트) 구현 스펙

> 기획: 페이블 / 구현: 오푸스
> 상품 상세 페이지에 과거 데이터 기반 롤링 백테스트를 붙이고,
> 손실확률을 계산해 상품 목록 컬럼으로도 노출한다.

## 확정된 규칙 (사용자 컨펌 완료)

1. **낙인(KI) 관찰**: 매일 종가 기준 상시관찰.
   발행일~만기 사이 어느 하루라도 워스트오브 자산의 종가가
   (기준가 × KI배리어%) 이하로 내려가면 낙인 발생으로 기록.

2. **만기 NoKI payoff**: 조기상환 없이 만기 도달 + 낙인 한 번도 미터치 →
   최대 쿠폰(만기 회차 연환산 수익률) 전액 지급. (손실 아님)

3. **손실확률 정의**: **조기상환이 한 번도 안 되고 만기까지 보유했는데
   손실이 확정된 경우**의 확률. 즉:
   - 모든 조기상환 평가일에서 워스트오브가 해당 회차 배리어 미달 (상환 실패)
   - 그리고 만기 시점에 손실 확정 (낙인 터치 + 만기 워스트오브가 원금 미만)
   - 이때 손실률 = 만기 워스트오브 수익률 (음수)

## 백테스트 방법론 (스크린샷 방식 재현)

- 기초자산: `core/market.py`의 TICKER_MAP으로 티커 해석, yfinance 과거 종가.
- **워스트오브**: 매 시점 여러 기초자산 중 (현재가/기준가) 비율이 가장 낮은 자산 기준.
- **롤링 표본**: 과거 데이터 구간에서 매 영업일을 "발행일"로 가정.
  각 발행일마다 기준가 = 그날 종가로 두고, 상품의 조기상환 스케줄
  (period_months × n, barriers_raw[n-1])대로 만기까지 전개 →
  1건의 표본(어느 회차에 상환됐는지 / 만기까지 갔는지 / 손실인지) 산출.
- 표본 수천 개 → 회차별 조기상환 빈도표 + 손실구간 분포표 집계.

## 데이터 기간 한계 (정직하게 표기)

- 20년(스크린샷 기준) 데이터가 없는 기초자산 많음(신규 상장 종목 등).
- 워스트오브 백테스트는 **모든 기초자산이 데이터를 가진 공통 구간**으로 제한됨.
- 화면에 반드시 표기: "표본기간 YYYY.MM ~ YYYY.MM, 표본 N개".
- 표본이 너무 적으면(예: < 200) "데이터 부족, 참고용" 경고 배지.
- 티커 매핑 실패한 기초자산이 하나라도 있으면 시뮬 불가 → "시뮬레이션 불가(기초자산 시세 없음)" 안내.

## 산출물

### 1. 상품 상세 페이지 "수익률 모의실험" 섹션
- 회차별 조기상환 표: [상환구분 | 수익률(연환산) | 발생횟수 | 발생빈도%]
- 손실 구간 분포표: [손실구간 | 발생횟수 | 빈도%] + 손실 소계
- 상단 요약: 조기상환 확률 합계 / **만기손실 확정 확률(=손실확률)** / 표본기간·수
- (선택) 워스트오브 자산 궤적 스파크라인 SVG

### 2. 상품 목록 손실확률 컬럼
- Product에 캐시 필드 추가: `loss_prob`(float, %), `sim_samples`(int), `sim_updated`(datetime).
- 배치(`simulate_products` 관리 커맨드)로 계산해 저장 — 매 요청마다 yfinance
  돌리면 느리므로 반드시 사전 계산·캐시.
- 주간청약/포트폴리오 목록 정렬 컬럼으로 노출. null(시뮬불가)은 "-".

## 구현 순서 제안
1. `core/backtest.py`: 순수 함수 `simulate(product, price_history) -> dict`
   (테스트 가능하게 시세를 인자로 주입, yfinance 호출과 분리)
2. 단위 검증: 인위적 가격 시나리오로 조기상환/낙인/손실 케이스 손계산과 대조
3. Product 필드 + 마이그레이션 + `simulate_products` 배치
4. 상세 페이지 섹션 + 목록 컬럼
5. 실데이터로 지수형 상품(KOSPI200 등 장기 데이터) 먼저 검증

## 주의
- 백테스트 로직은 **틀려도 그럴듯해 보인다**. 반드시 손계산 단위테스트로 검증 후 화면에 노출할 것.
- 세전/세후: 표는 세전 수익률(스크린샷과 동일). 세후는 기존 포트폴리오 계산과 중복이므로 여기선 세전만.

# 함께 처리할 데이터 버그 픽스 (2026-07-18 발견, 페이블 진단 완료)

## 픽스 1 — NH투자증권 006-알파벳 상품번호 중복 (24건)
- **원인**: NH의 KOFIA 상품명은 `"N2 ELS 369"`, `"NH투자증권(ELS) 24981"`처럼 회차번호가
  **끝에** 붙고 "제N회" 형식이 아님. `core/kofia_scraper.py`의
  `product_no = m.group(1) if m else _v(el, 22)[-6:]` 에서 "제N회" 정규식이 실패 →
  product_code 뒤 6자리("006QE1")로 폴백 → exe엑셀의 정상 상품(product_no=369, product_code='')과
  매칭 실패해 별도 행 생성 = 중복.
- **수정**: kofia_scraper의 product_no 추출을 개선.
  "제N회" 실패 시 **상품명 끝의 숫자 토큰**을 추출 (예: `"N2 ELS 369"` → `369`,
  `"NH투자증권(ELS) 24981"` → `24981`). 이러면 scrape_kofia upsert가
  (issuer, product_no, sub_end)로 exe 행을 찾아 병합(product_code 세팅)함.
  마지막 폴백(코드 6자리)은 그래도 숫자 못 찾을 때만.
- **정리**: 위 수정 후, 기존 006-접두 24건 삭제.
  `Product.objects.filter(issuer='NH투자증권', product_no__regex=r'^006[A-Z]').delete()`
  삭제 전 개수 확인하고, 삭제 후 scrape_kofia 재실행해서 정상 병합·재발 없음 검증.
  (다른 발행사도 상품명이 "제N회" 아닌 곳 있는지 같이 점검 — 삼성/신한 등)

## 픽스 2 — 한국투자증권 656형 (상승참여형) 노낙인·배리어 누락
- 설명 예: `[온라인 전용상품]기초자산:삼성전자/9개월만기 최초4개월(1회)이후 만기관찰/
  조기상환: 85이상(4개월, abs 20.00%(연 60.00%))/만기시 100 이상 상승참여율 200%, 100 미만 하락참여율 100%`
- **NoKI**: 현재 파서(`상승참여율` 룰)로 재적용하면 이미 NoKI로 나옴. 문제는
  이 행이 **exe 소스**(product_code='')라 옛 파서 결과(KI=None)가 방치된 것. import_els는 새 파일만 처리.
- **배리어**: `조기상환: 85이상(...)` 형식 미지원 → 1차 배리어 85 추출 패턴 추가.
  `core/parsers.py` extract_barriers에 `r'조기상환:\s*(\d+)\s*이상'` 폴백 추가(단일 배리어).
- **재파싱 커맨드**: `reparse_products` 관리 커맨드 신규 —
  모든 Product를 순회하며 description으로 extract_ki/barriers/period/classify_asset
  재계산해 KI/is_no_ki/barrier_first/last/barriers_raw/period_months/asset_type 갱신.
  (scrape_kofia는 매 실행 재파싱되지만 exe 소스 3800여 건은 방치 상태이므로 일괄 정리 필요)
- 재파싱 후 exe소스 KI/배리어 커버리지도 함께 상승할 것 — 회귀 없는지 수치 확인.
